# 进度追踪实践

> 本工作区进度追踪（WORKLOG/PROJECT_STATUS/CURRENT_TASK/Backlog/接力机制）的本地工作流总结，涵盖工作区级、项目级、智能体级 3 层进度管理体系。

## 当前工具链

### 工作区级（d:\xiangmu\）
- **WORKLOG.md** — 工作区级工作日志（最新记录在顶部，倒序排列）
  - 滚动归档：保留最近 50 条/3 个月，旧记录归档到 `worklog-archive/`
  - 格式：`## YYYY-MM-DD（HH:MM）— 主题（第 N 会话）`
  - 内容：决策记录/任务/执行步骤/产出/状态/关键决策/下一步
  - 现状：5 会话完整记录（2026-06-19 第 1 会话 → 2026-06-20 第 5 会话）
- **CURRENT_TASK.md** — 当前任务接力文件
  - 结构：任务 + 步骤进度表（步骤/状态/摘要）
  - 用途：跨智能体接力（任一智能体读此文件即可知当前进展）
  - 现状：多智能体协同系统搭建，4.8 完成，5+6 待启动
- **MANUAL_CLEANUP.md** — 手动清理清单
  - 记录 Trae 无法直接删除的目录（被进程锁定）
  - 步骤 1-3 完整流程（关闭进程 → PowerShell 删除 → 长路径终极方案）

### 项目级（每个项目根目录）
- **PROJECT_STATUS.md** — 项目活档案（25+ 项目已建）
  - 必填段：基本信息/运行位置/后端与依赖服务/仓库/开发阶段/当前进展/已知问题/关键决策记录
  - 接力机制：会话开始必读，会话结束必更新（合并更新不覆盖用户手写内容）
  - 密钥不入档只记密钥位置
  - **规则文件**：`d:\xiangmu\_meta\rules\project-status.md`
- **WORKLOG.md**（项目级，可选）— SuanJia/stock-sim/xunbiao/TradingAgents 等活跃项目有
  - 项目内事件记录（按日/按会话）
  - 滚动归档：>50 条或最早记录早于 3 个月

### 智能体级（接力机制）
- **接力机制 3 件套**：
  - `WORKLOG.md` — 历史事件（最新在顶）
  - `PROJECT_STATUS.md` — 当前状态（活档案）
  - `CURRENT_TASK.md` — 当前任务步骤进度
- **会话开始流程**（6 步）：
  1. 读 WORKLOG.md 最近 3 条
  2. 读 PROJECT_STATUS.md（运行位置/后端/阶段/下一步）
  3. 检查 WORKLOG 是否需归档（>50 条或最早记录早于 3 个月）
  4. 若 CURRENT_TASK.md 状态"进行中"先问用户是否继续
  5. 检查是否有未完成 relay（接力标记）
  6. 用户给任务后，开始执行
- **会话结束流程**（3 步）：
  1. WORKLOG.md 顶部追加记录
  2. 更新 PROJECT_STATUS.md（合并更新不覆盖）
  3. CURRENT_TASK.md 完成则改"已完成"，未完成则更新"下一步"
- **时间格式**：YYYY-MM-DD HH:MM，Asia/Shanghai，精确到分钟
- **规则文件**：`d:\xiangmu\_meta\rules\relay.md`

- **测试方式**：5 智能体（Trae/Codex/OpenCode/Hermes/Claude Code）独立读 WORKLOG.md + PROJECT_STATUS.md + AGENTS.md

## 痛点

1. **3 层接力有重叠**：工作区级 WORKLOG.md + 项目级 WORKLOG.md + CURRENT_TASK.md 三处可能冲突
2. **CURRENT_TASK.md 大幅过时**：2026-06-20 第 4 会话才重写，之前 4-4.8 步骤全无
3. **PROJECT_STATUS.md 维护成本**：25+ 项目每次会话结束都要更新，靠自觉容易漏
4. **夜间任务结果散落**：night-tasks/results/ 多个 .jsonl + .log + .md 混合，缺统一索引
5. **Backlog 无优先级**：SuanJia/docs/backlog.md 是 TODO 列表，无 P0/P1/P2 标记
6. **Cron 工具分散**：Hermes Cron / GitHub Actions / Task Scheduler / 飞书机器人 4 套调度系统，无统一面板
7. **接力标记缺失**：智能体间无明确 "我正在做 X，X 完成请接 Y" 的标记，靠 PROJECT_STATUS 下一步推断
8. **5 智能体无总览**：5 智能体各自跑，无 dashboard 展示谁在做什么
9. **风险跟踪静态**：risk_registry.json 写入后无定期复审机制
10. **里程碑无提醒**：stock-sim "7 月 1 日双月频调仓日"靠 WORKLOG 记录，无日历/提醒
11. **Backlog 与 WORKLOG 脱钩**：backlog.md 的 TODO 不知道谁在负责、什么时间完成
12. **手动清理人工**：MANUAL_CLEANUP.md 步骤 1-3 全部人工，~575 MB 释放靠手动

## 优化方向（结合 AI Agent）

### 1. 进度仪表盘（实时聚合）
- **方案**：建 `d:\xiangmu\信息\项目仪表盘.md`
  - 拉取所有 PROJECT_STATUS.md 的"当前阶段"+"当前进展"段
  - 按 stage 分类（前期跟踪/招标中/已中标/执行中/已完成）
  - 风险项目标红（基于 risk_registry.json）
  - 飞书每日推送（Hermes Cron）
- **AI 角色**：
  - Claude Code 知识管家负责聚合（扫 25+ PROJECT_STATUS.md）
  - Hermes Cron 每日 9:00 触发
  - 飞书 Webhook 推送到"项目监理群"

### 2. 3 层接力标准化
- **方案**：明确 3 层边界
  - **工作区级 WORKLOG**：跨项目事项 + 智能体协作记录
  - **项目级 WORKLOG**：单项目内技术细节
  - **CURRENT_TASK.md**：单项目当前任务步骤
  - **PROJECT_STATUS.md**：跨层状态总览（永远在最新状态）
- **规则更新**：`_meta/rules/relay.md` 加 3 层边界定义
- **AI 角色**：Claude Code 知识管家负责规则维护 + 智能体执行时检查

### 3. Backlog 升级（GitHub Issues 镜像）
- **方案**：
  - SuanJia/docs/backlog.md 拆分到 GitHub Issues（label: P0/P1/P2/area:xxx）
  - 自动同步：本地 .md → GitHub Issues（gh CLI）
  - 反向：GitHub Issues 关闭 → 本地 .md 删除条目
- **AI 角色**：
  - Codex：扫描代码 TODO → 自动建 Issue
  - Claude Code：本地/远端同步
  - Trae：移动端查看/编辑 Issue

### 4. 5 智能体总控面板
- **方案**：建 `d:\xiangmu\CURRENT_AGENT_STATUS.md`
  - 每智能体状态：空闲/工作中/等待输入/错误
  - 当前任务：智能体 → 任务 → 进度
  - 飞书实时推送（lark-im 群消息）
- **AI 角色**：
  - Claude Code 知识管家：聚合状态
  - 各智能体：每 5 分钟心跳上报
  - Hermes Cron：聚合 + 推送

### 5. 接力标记协议（Handoff Protocol）
- **方案**：建 `.handoff/` 目录
  - `intent.md`（当前意图）
  - `agent.md`（当前执行智能体）
  - `progress.md`（执行进度）
  - `next-step.md`（下一步建议）
  - `blocker.md`（阻塞问题）
- **使用**：智能体开始任务 → 创建 .handoff/；中途换智能体 → 新智能体读 .handoff/
- **AI 角色**：所有智能体遵循协议

### 6. 里程碑 + 日历提醒
- **方案**：
  - 关键里程碑写入 `信息/05-飞书配置.md` 旁路
