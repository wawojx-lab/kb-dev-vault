# 项目：建设聚合模型平台

> **目标**：把 5 智能体（Trae/Codex/Claude Code/Hermes/OpenCode）+ 多 LLM 模型统一接入
> **总工时**：13 小时
> **核心选型**：Strands SDK + Bedrock AgentCore + LangGraph

## 背景与动机

### 当前痛点
- 5 个 AI 智能体各自独立调用模型 API，**重复造轮子**
- 模型选型缺乏统一基准（A/B 对比靠人肉）
- 安全策略分散（每个智能体一套密钥管理）
- 成本不可见（不知道哪个项目花了多少 token）

### 目标效果
- **统一入口**：所有智能体通过聚合平台调模型
- **智能路由**：根据任务类型自动选最优模型（编码/推理/对话）
- **成本可见**：每个项目 / 每个用户的 token 消耗面板
- **安全统一**：IAM 集中授权 + CloudTrail 集中审计

## 13 小时分阶段拆解

### Phase 1：需求分析 + 选型（2h）✅ 关键决策

**目标**：确定技术栈 + 业务边界

| 任务 | 时间 | 产出 |
|------|------|------|
| 调研 5 智能体调用模式 | 0.5h | 调用模式矩阵 |
| 盘点现有 API 密钥和成本 | 0.5h | 密钥清单 + 月度账单 |
| Bedrock vs Azure AI 选型 | 0.5h | 选型决策文档 |
| 定义路由策略 | 0.5h | 路由规则表 |

**关键决策**：
- 主力模型：**Bedrock Claude Sonnet 4**（编码 + 推理）
- 实验模型：**Bedrock Nova Pro**（成本敏感场景）
- 编排层：**Strands SDK**（Python 优先，多模型中立）
- 部署：**AgentCore Runtime**（生产）+ 本地 Strands（开发）

**应用案例**：[comparisons/bedrock-vs-azure-ai.md](../comparisons/bedrock-vs-azure-ai.md)

### Phase 2：模型路由层（3h）

**目标**：实现"调用方不感知模型" + 智能路由

```python
# 路由层接口设计
class ModelRouter:
    def route(self, task: Task) -> Model:
        # 编码任务 → Claude Sonnet 4
        # 长文摘要 → Claude Haiku（成本低）
        # 中文对话 → Qwen 72B
        # 推理/数学 → Claude Opus 4
        pass
```

**3 个子任务**：
- 2.1 抽象模型接口（1h）：统一 `invoke()` / `stream()` 签名
- 2.2 实现 4 模型适配器（1.5h）：Bedrock / OpenAI / 自托管 vLLM / Anthropic 直连
- 2.3 路由规则引擎（0.5h）：基于任务类型 + 成本 + 延迟的决策

**验收**：
- [ ] 5 智能体全部通过新路由层调用
- [ ] 路由决策可观测（每次路由记录原因）
- [ ] 单元测试覆盖 4 模型适配器

### Phase 3：智能体编排层（4h）— 核心

**目标**：把 5 智能体接入统一编排框架

**选型**：**LangGraph**（多 Agent 协作）+ Strands（单 Agent 快速开发）双层

```
┌─────────────────────────────────────┐
│ LangGraph Supervisor (多 Agent 协调) │
├─────────────────────────────────────┤
│ Trae / Codex / Claude Code / Hermes │
│ (Strands 包装为子 Agent)              │
├─────────────────────────────────────┤
│ 共享工具集（MCP Server）              │
│ - 知识库检索                          │
│ - 股票数据 API                        │
│ - Git 操作                            │
└─────────────────────────────────────┘
```

**4 个子任务**：
- 3.1 Strands 包装现有 5 智能体为可调用单元（1.5h）
- 3.2 共享 MCP Server 搭建（1h）：知识库 + 业务 API
- 3.3 LangGraph Supervisor 编排（1h）：决策"哪个 Agent 处理"
- 3.4 端到端测试（0.5h）：跨智能体协作跑通

**应用案例**：[concepts/企业Agent集成.md](../concepts/企业Agent集成.md) + [concepts/工作流图.md](../concepts/工作流图.md)

### Phase 4：安全边界（2h）

**目标**：满足企业生产级安全基线

**应用案例**：[concepts/安全边界模式.md](../concepts/安全边界模式.md) 的 L2 标准级

| 措施 | 时间 | 实施 |
|------|------|------|
| IAM 角色 + 最小权限 | 0.5h | 每智能体独立 Role |
| VPC + PrivateLink | 0.5h | 私有子网部署 |
| KMS 加密 + Secrets | 0.5h | 静态/传输双加密 |
| CloudTrail 审计 | 0.5h | 全量调用链记录 |

**验收**：
- [ ] 安全检查清单 10 项全部通过（参考安全边界模式文档）
- [ ] 智能体无公网 IP
- [ ] CloudTrail 日志可在 S3 持久保留 90 天

### Phase 5：监控 + 告警（2h）

**目标**：生产可观测性 + 异常告警

| 维度 | 工具 | 指标 |
|------|------|------|
| Token 成本 | CloudWatch + 自建 dashboard | 按项目/用户/模型分桶 |
| 调用延迟 | OpenTelemetry → CloudWatch | P50 / P95 / P99 |
| 错误率 | CloudWatch Alarms | >1% 触发 |
| 异常调用 | CloudTrail Lake | 短时间大量删除/外发 |
| 模型质量 | Bedrock Evaluations | 准确性 + 安全性评分 |

**5 个子任务**：
- 5.1 CloudWatch 面板搭建（0.5h）
- 5.2 OpenTelemetry trace 接入（0.5h）
- 5.3 异常告警规则（0.5h）
- 5.4 成本分摊报表（0.5h）：按项目 / 智能体维度
