# 任务拆解方法论

基于多项目实战的复杂任务拆解方法。

## 4 层拆解

### 第 1 层：阶段
- 调研 / 设计 / 实现 / 测试 / 部署

### 第 2 层：模块
- 前端 / 后端 / 数据库 / 部署

### 第 3 层：用户故事
- "作为 XX，我想 YY，这样 ZZ"
- 每个故事 = 1 个可发布的功能

### 第 4 层：原子任务
- 1-2 小时工作量
- 单一职责
- 有 done 定义
- 可独立 commit

## 5 个拆解原则

1. **粒度适中**：1-2 小时（不是 1 分钟也不是 1 天）
2. **依赖清晰**：DAG 表达，不是列表
3. **可验证**：done 定义明确
4. **可交接**：每个任务有"如何让别人接手"
5. **可回退**：每个里程碑有 tag / 分支

## 验收清单

每个子任务必填：

- [ ] **输入**：开始时需要什么（前置任务 / 数据 / 工具）
- [ ] **输出**：完成时交付什么（代码 / 文档 / 数据）
- [ ] **done 定义**：怎样算完成
- [ ] **失败回退**：失败时怎么办
- [ ] **接手人**：谁可以接手（其他智能体）
