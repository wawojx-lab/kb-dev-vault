# 知识库下一步调研（五方向详细）

> 调研时间：2026-06-20
> 调研目的：补完知识库功能与治理能力，规划后续落地

## 方向 1：知识图谱构建（高优先级）

### 问题
OKF 格式只规定单文件结构，知识之间的关系依赖 Markdown 链接。如何系统化构建知识图谱？

### 已有能力盘点

**flywheel 已有**（[src/kb/graph.py](file:///d:/xiangmu/_kb_flywheel/src/kb/graph.py)）：
- `kb_graph_viz` 工具：Mermaid 图表导出
- `kb_affected_pages`：反向链接索引
- `kb_evolve`：分析知识缺口，建议新连接

**memory MCP 已有**（已配）：
- 实体-关系-观察（entity-relation-observation）三元组
- 9 个工具：创建/搜索/打开/读取图谱/删除
- 缺点：图谱数据是**独立的** JSON 存储，不与 OKF 文件互通

**memory-engineering MCP 已有**（已配）：
- 在 memory 基础上加了 capture_learning/capture_correction/add_pattern
- 9 个工具，针对工程知识优化
- 同样：独立存储

### 决策树

```
要建知识图谱？
├── 关系简单（项目-状态-技术栈）→ OKF Markdown 链接 + flywheel kb_graph_viz
├── 关系复杂（多维关联）→ memory MCP（独立图谱）
└── 工程知识 + 学习记录 → memory-engineering MCP
```

### 落地建议

1. **短期**：用 flywheel `kb_graph_viz` 验证 OKF 链接图谱（先 stock-sim/SuanJia 主题试跑）
2. **中期**：评估 memory MCP 是否需要接入（如果需要复杂图谱查询）
3. **避免**：不要双写（OKF + memory 同步），选一个主存

### 实操方法（待你确认）

在 Claude Code 里执行：
```
用 kb_graph_viz 工具，生成 stock-sim 主题的 Mermaid 图谱
```

## 方向 2：知识治理流程（高优先级）

### 问题
所有智能体可写后，如何保证知识库质量？

### flywheel lint 能力

工具：`kb_lint`
实测（Codex 试跑）：121 问题（10 errors / 1 warning / 110 info）
- errors：broken wikilinks（demo 数据问题，工具本身正常）
- warning：wikilink cycle detection 超时
- info：frontmatter 日期过期 + wikilink 循环

### 治理流程设计

**每日治理**（自动/手动触发）：
- `kb_lint`：发现新问题
- `kb_save_lint_verdict`：保存判定结果
- `kb_refine_list_stale` / `kb_refine_sweep`：清理过期

**每周治理**（手动，建议周末）：
- `kb_stats`：看知识库体量
- `kb_verdict_trends`：看质量趋势
- `kb_evolve`：找知识缺口
- `kb_detect_drift`：查因原始源变更而可能过期的页面

**每月治理**（手动）：
- `kb_graph_viz`：可视化整库，检查孤立节点
- `kb_reliability_map`：查信任评分低的页面

### 角色分工

| 角色 | 职责 |
|------|------|
| 用户 | 审批大改动、最终质量把关 |
| 知识管家（Claude Code + flywheel） | 执行 lint/compile/evolve、生成报告 |
| 其他智能体 | 写笔记时遵守 OKF 格式 |

### 关键决策
- **不强制每次写都 lint**（成本太高）——改为定期 lint + 重要提交前 lint
- **lint 报告存到 _kb/raw/lint-reports/**：作为治理历史
- **eolve 建议**作为知识管家的"待办"，用户决定采纳与否

## 方向 3：Dynamic Workflows 实战（中优先级）

### 问题
何时用 Dynamic Workflows，何时用 flywheel？

### 关键发现

**Dynamic Workflows 核心**：AI 临场写 JavaScript 编排脚本（[Claude Code v2.1.154+](https://docs.claude.com/en/docs/claude-code/dynamic-workflows)，2026-05-28）

**实测案例**（来自调研）：
- 推文 10000+ 点赞 330 万浏览：spins up large fleet of coordinated subagents in parallel
- 75 万行 Zig→Rust 迁移：11 天
- `goal` 模式 75 小时翻车 → 才看懂 dynamic workflows
- Sisyphus Labs 公开指控 Anthropic 抄袭 OMO

### flywheel vs Dynamic Workflows 决策树

```
要做的任务？
├── 单次知识编译/查询/lint → flywheel（专用工具，标准化）
├── 跨多个主题大规模研究/迁移 → Dynamic Workflows（编排自由）
├── 临时小修补/紧急查询 → flywheel kb_query（快速）
└── 探索性多角度决策 → Dynamic Workflows（强制多立场）
```

### 何时不用

- 简单一两步操作
- 需频繁用户拍板
- 高风险改动（金融交易、删除操作）

### 落地建议

1. **准备场景**：用 stock-sim 的"下一步要做什么"作为 Dynamic Workflows 试跑
2. **对比实验**：同样任务用 flywheel 跑 vs Dynamic Workflows 跑，对比输出质量
3. **成本控制**：设 max-iterations 和 max-tokens 上限

### 备注
- Dynamic Workflows 是 research preview（官方标记）
- 强依赖 Opus 4.8 模型（其他模型可能不稳定）
- 中间过程不占主上下文（脚本变量流转）= 节省 token

## 方向 4：远程访问方案（中优先级）

### 问题
知识库如何在多设备/远程访问？

### 方案调研（2026 年最新）

| 方案 | 成本 | 隐私 | 速度 | 复杂度 | 适用场景 |
|------|------|------|------|--------|---------|
| **Obsidian 官方 Sync** | $4-8/月 | 云端加密 | 快 | 零 | 个人多设备 |
| **Self-hosted LiveSync** | 免费 | 自托管 | 中 | 中 | 隐私敏感 |
| **Syncthing + Git** | 免费 | 完全私密 | 中 | 中 | 多设备 + 版本回退 |
| **Git 私有仓库** | 免费 | 私有 | 中 | 低 | 单一远程设备 |
| **iCloud/OneDrive/坚果云** | 已有 | 依赖服务商 | 中 | 零 | 已有云盘用户 |

### 推荐方案

**主力**：**Git 私有仓库**（`d:\xiangmu\_kb\.git` + 远程 **GitHub**）
- 已有 Git 规范（`d:\xiangmu\_meta\rules\git.md`）
- 完全免费 + 完全可控（GitHub 私有仓）
- 改动有版本历史
- 缺点：非实时同步（需要 commit + push/pull）

**辅助**：**Syncthing**（如果你有 NAS 或第二台电脑）
- 实时同步
- 不依赖云服务
- 配合 Git 做版本回退

**避免**：
- Obsidian 官方 Sync（每月 $4-8 不必要）
- iCloud（已有加密但 Windows 体验差）

### 决策记录（2026-06-20）
- **Git 仓库**：GitHub 私有仓
- **飞书同步**：暂不同步

### 落地步骤
1. 在 `d:\xiangmu\_kb\` 初始化 Git 仓库
2. 加 .gitignore（排除 _kb_flywheel/、敏感文件）
3. 推送到 Gitea 私有仓库
4. 远程设备 clone + pull 即可

## 方向 5：跨智能体知识共享（中优先级）

### 问题
Trae/Codex/OpenCode/Hermes/Claude Code 如何共享知识？

### 现状盘点

| 智能体 | 文件系统 | obsidian MCP | memory MCP | kb MCP | 知识库可写 |
|--------|:---:|:---:|:---:|:---:|:---:|
| Trae | ✅ | — | — | — | ✅ |
| Codex | ✅ | — | — | — | ✅ |
| OpenCode | ✅ | — | — | — | ✅ |
| Hermes | ✅ | — | — | — | ✅ |
| Claude Code (CLI) | ✅ | ✅ | ✅ | ✅ | ✅ |
| Claude Code (IDE) | ✅ | ✅ | ✅ | ✅ | ✅ |

### 知识共享机制设计

**主存**：`_kb/` 文件系统（所有智能体都能直接读写）

**单点写入**（不推荐，已废除）：知识管家

**多源写入**（当前）：
- 所有智能体直接写 `_kb/raw/`、`_kb/wiki/`、`_kb/research/`、`_kb/projects/`
- 遵循 OKF 格式（type 字段 + frontmatter + Markdown 链接）
- 知识管家（Claude Code）负责 lint/compile/evolve

### 冲突处理

- **OKF 文件级锁**：基于 frontmatter 的 timestamp 字段
- **人工仲裁**：所有智能体写完的笔记，由知识管家定期 lint，发现冲突报告
- **Git 版本控制**：每次写都 commit，可回退

### memory MCP 的特殊性

memory MCP 是**独立图谱存储**（JSON 格式），不是 Markdown 文件：
- 只 Claude Code 能访问
- 适合临时快速记录（不需 OKF 格式）
- 长期沉淀应迁到 _kb/wiki/

### 落地建议
1. **短期**：所有智能体写 _kb 时遵循 OKF 格式
2. **中期**：在 _meta/AGENTS.md 加"跨智能体知识共享"章节
3. **长期**：评估 memory MCP 是否需要桥接到 OKF（目前不必要）
