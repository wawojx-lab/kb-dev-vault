# 知识库下一步调研执行计划

> 时间：2026-06-20
> 目标：完善知识库体系，调研行业最佳实践，规划落地路径
> 状态：待执行

## 背景

知识库基础架构已搭建完成：
- `_kb/` 开发域（raw/wiki/research/projects 四层 + OKF 格式）
- `_kb_personal/` 个人域（4 目录 + 5 文件）
- flywheel 引擎（_kb_flywheel/）已 clone + install
- kb MCP 待配置（步骤 5）

接下来的调研重点是**补完知识库的功能和治理能力**。

## 待调研方向

### 1. 知识图谱构建（优先级：高）
**问题**：OKF 格式只规定单文件结构，知识之间的关系依赖 Markdown 链接。如何系统化构建知识图谱？

**调研内容**：
- Memory MCP（Claude Code 已配）：实体-关系-观察的图谱结构
- flywheel 的 graph 功能：`kb_graph_viz`（Mermaid 导出）
- 现有工具对比：logseq/dendron/roam 的图谱能力
- 实践：先从 stock-sim/SuanJia 主题构建小规模图谱试跑

### 2. 知识治理流程（优先级：高）
**问题**：所有智能体可写后，如何保证知识库质量？

**调研内容**：
- flywheel 的 lint 规则细节：121 个问题的分类（errors/warnings/info）
- kb_evolve 工具的演进策略
- 知识管家的定期审计频率和触发条件
- 用户/智能体/管家的三方协作流程

### 3. Dynamic Workflows 实战（优先级：中）
**问题**：何时用 Dynamic Workflows，何时用 flywheel？

**调研内容**：
- 实测：用一个研究任务试跑 `/deep-research`
- 对比：flywheel kb_query vs Dynamic Workflows 的输出差异
- 成本：token 消耗对比
- 决策树：什么场景用哪个

### 4. 远程访问方案（优先级：中）
**问题**：知识库如何在多设备/远程访问？

**调研内容**：
- Obsidian 同步方案（官方 Sync/Syncthing/Git）
- 飞书知识库对接（lark-wiki skill）
- Hermes 远程接入的完整链路
- 个人域的远程访问权限控制

### 5. 跨智能体知识共享（优先级：中）
**问题**：Trae/Codex/OpenCode/Hermes/Claude Code 如何共享知识？

**调研内容**：
- 现有 obsidian MCP（已配）的搜索能力
- 各智能体的文件系统工具能力
- memory MCP（Claude Code 独占）的图谱数据如何导出
- 知识库作为"单一事实源"的实现路径

## 落地顺序

1. **立即**：配置 kb MCP + 验证 28 工具
2. **本周**：方向 1（知识图谱）+ 方向 2（治理流程）
3. **下周**：方向 3（Dynamic Workflows 实战）
4. **后续**：方向 4（远程访问）+ 方向 5（跨智能体共享）
