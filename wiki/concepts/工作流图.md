# 可追踪工作流图（Traceable Workflow Graph）

用**有状态有向图**表达 AI Agent 工作流的方法。每个节点是计算步骤，每条边是控制流转移，全局 state 跨节点持久化。

对比传统 DAG（见 langgraph-vs-dag 对比页），可追踪工作流图多了**状态**和**可恢复性**。

## 核心要素

| 要素 | 说明 |
|------|------|
| **State** | 带 schema 的共享状态（TypedDict/Pydantic），Reducer 控制合并策略 |
| **Node** | 同步/异步函数，接收 state → 计算 → 返回更新 |
| **Edge** | 普通边（确定路由）+ 条件边（函数判断）+ 动态边（Command/Send） |
| **Checkpoint** | 每 super-step 后自动保存 state，实现时间旅行和故障恢复 |
| **Super-step** | 单轮图迭代，可包含多个并行节点 |
| **Interrupt** | 暂停执行，等待外部输入后恢复 |

## 与传统 DAG 的关键区别

- **DAG**：数据流导向，节点间传递数据，无状态或外界状态
- **工作流图**：状态导向，所有节点读写同一 State，Checkpointer 冻结状态历史
- **DAG**：拓扑排序后一次性执行
- **工作流图**：循环+条件+中断，可长时间运行、暂停、恢复

## 可追踪性设计

### 1. 每个 transition 都有 checkpoint

LangGraph 每 super-step 自动保存 state → 形成 checkpoint 链 → 可在时间轴上任一点回放。

```python
# 获取历史 checkpoint
state_before = graph.get_state(config, checkpoint_id)
# 从任意 checkpoint 恢复
graph.invoke(None, {"configurable": {"thread_id": tid, "checkpoint_id": cid}})
```

### 2. 每个节点都被 traced

LangSmith 自动捕获每节点的输入/输出/耗时/错误。可通过 `config["metadata"]` 访问运行时元数据（step / node / triggers / path）。

### 3. 状态变更可审计

每步 state diff 可提取 + 还原 → 完整审计日志。

## 多节点编排模式

| 模式 | 描述 | LangGraph 实现 |
|------|------|----------------|
| **链式** | A → B → C | `add_edge("A","B"); add_edge("B","C")` |
| **路由** | 条件分支 | `add_conditional_edges("A", router)` |
| **并行** | 扇出多节点 | 多出边自动并行执行 |
| **Map-Reduce** | 动态创建 N 个节点 | `Send("worker", state_i)` |
| **循环** | 反复执行直到终止 | `add_edge("tool", "llm")` 形成回路 |
| **嵌套** | 子图封装 | `add_node("sub", subgraph.compile())` |
| **审批** | 暂停等待人 | `interrupt()` + `Command(resume=...)` |

## 与接力机制的映射

| 接力机制 | 工作流图对应 |
|---------|-------------|
| 接力文件（PROJECT_STATUS.md） | Checkpoint（持久化 state） |
| 必读 3 件套 | State 初始化 + Config（thread_id） |
| 必写 2 件套 | Node 输出 + Checkpointer 自动保存 |
| 跨智能体接力 | Command.PARENT + Subgraph handoff |

## 何时用

- 需要**长时间运行**的 Agent 任务（> 1 分钟）
- 需要**人类审批**的关键节点
- 需要**故障恢复**的生产环境
- 需要**完整审计追踪**的合规场景
- 涉及**多 Agent 协作**的复杂工作流
