## 调研 1：Google Open Knowledge Format (OKF)

- 发布：2026-06-16
- 来源：51cto.com / 微信公众号"AI寒武纪"
- 仓库：https://github.com/GoogleCloudPlatform/knowledge-catalog/tree/main/okf

### 核心概念
- 一个 OKF bundle = 一个 Markdown 文件目录
- 每个文件代表一个概念（数据表/数据集/指标/API/操作手册）
- 文件路径 = 概念的唯一标识
- 每个文件 = YAML frontmatter（结构化字段） + Markdown 正文（自由格式）
- 概念之间用普通 Markdown 链接互相引用 → 关系图

### 三个设计原则
1. 最小约束：只强制 `type` 字段，其他全自由
2. 生产者消费者独立：人写的 AI 能读，AI 生成的 bundle 另一个 AI 能查
3. 格式不是平台：不绑定云服务/SDK/账号

### 三个参考实现
- 数据丰富 agent（BigQuery 自动生成 OKF 概念文档）
- 静态 HTML 可视化工具（无后端/无安装/数据不离开本地）
- 三个样例 bundle（GA4 电商/Stack Overflow/比特币公开数据）

## 调研 2：Claude Code Dynamic Workflows

- 发布：2026-05-28
- 来源：CSDN / 公众号
- 团队：Sisyphus Labs 公开指控 Anthropic 抄袭 OMO

### 核心机制
- AI 临场写 JavaScript 编排脚本
- 后台跑，派生 subagent 并行干活
- 中间结果在脚本变量里流转，不回到主对话
- 适合：代码库批量扫描/大规模迁移/研究类/内容生产/多角度决策
- 不适合：一两步小修补/需频繁拍板/高风险改动

### 与 subagent/skill 对比
| 方式 | 谁握计划 | 中间结果 | 上下文占用 |
|------|---------|---------|-----------|
| Subagent | Claude 逐轮判断 | 回主对话 | 高 |
| Skill | Claude 按 SOP 走 | 回主对话 | 高 |
| Dynamic Workflow | 脚本自己决定 | 脚本变量 | 低 |

### 实战案例
- 代码库四维扫描：架构/安全/性能/测试并行
- Bun 作者 Zig→Rust 11 天 75 万行
- /deep-research 内置命令

## 调研 3：flywheel（LLM-wiki 知识编译引擎）

- 仓库：https://github.com/Asun28/llm-wiki-flywheel
- 工具数：官方 28，实测 24-27（差异可能是版本）
- MCP 启动：Claude Code 模式默认 LLM（不需要 API Key）
- CLI 模式（kb compile）需要 ANTHROPIC_API_KEY
- 与 obsidian MCP 不冲突（obsidian=文件 IO，flywheel=知识编译）

## 本地集成方案（四层架构）

- 格式层：OKF v0.1
- 隔离层：开发域 _kb/ + 个人域 _kb_personal/
- 写入层：所有智能体可写（取消单点写入）
- 编译层：flywheel + Dynamic Workflows

## 实测数据

- flywheel 试跑（Codex 执行）：lint 通过，121 问题（全 demo 数据）
- Claude Code CLI kb MCP：24 个工具正常加载
- 配置位置：`~/.claude.json` 全局 mcpServers（不在项目 .mcp.json）
- 路径转义问题：`claude mcp add -e` 吞反斜杠和下划线 → 用 `cmd /c bat` 包装
